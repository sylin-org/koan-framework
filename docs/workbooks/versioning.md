# Versioning

Operate the ARCH-0082 two-tier versioning system. Bump package versions, decide what should bump, recover when versions go sideways.

> Companion ADR: [ARCH-0082](../decisions/ARCH-0082-versioning-strategy.md) — read this if you want to know *why* the system is shaped the way it is.
> Companion workbook: [nuget-publishing.md](nuget-publishing.md) — once versions are decided, how they reach nuget.org.

---

## When to use this

- You're preparing a commit and want to know what version it'll produce
- A merge to main is queued and you want to preview the bump
- A package bumped (or didn't bump) and you want to understand why
- You're adding a new package and need to wire it into the versioning system
- Versions look wrong on main and you need to fix them

**Prerequisites:**

- PowerShell 7+ (`pwsh`)
- Git, .NET 10 SDK
- Working tree at the repo root
- For local computations: nothing else
- For releases: see [nuget-publishing.md](nuget-publishing.md) — releases happen via CI

---

## Mental model (30 seconds)

Every NuGet-packable project under `src/` is one of two kinds:

- **Kernel** (14 packages) — contracts that other packages compile against. They share one version (`$(KoanKernelVersion)`) and move together.
- **Periphery** (everything else, ~90 packages) — implementations, adapters, connectors. Each has its own version. Each bumps only when its own folder has new commits since the last release tag.

The kernel manifest lives at [build/kernel-manifest.txt](../../build/kernel-manifest.txt). The current versions live at [build/versions.props](../../build/versions.props) — generated by tooling, never hand-edited.

Bump magnitudes follow conventional commits:

- `feat!:` or `BREAKING CHANGE:` in commit body → **major**
- `feat:` → **minor**
- everything else (`fix:`, `refactor:`, `perf:`, `chore:`, etc.) → **patch**
- no commits touching the folder → **no bump**

Releases are tagged `release/v<kernel-version>` (e.g., `release/v0.8.0`). The tag marks the commit that recorded those per-package versions in versions.props.

That's the whole model. Everything else in this workbook is just how to operate on it.

---

## Happy path

You don't operate the versioning system on every commit. The CI workflow does it on every merge to `main` — see [nuget-publishing.md](nuget-publishing.md) for that flow.

For *previewing* what would happen, you have one command:

```pwsh
pwsh scripts/versioning/Show-VersionStatus.ps1
```

Output tells you:

- Which packages would bump
- Why (commit count + magnitude)
- The reference tag the script compared against

Read it. If the bumps match your intent, you're done — push your branch and let CI do the rest.

### Periphery hotfix

Most common case. You changed code in `src/Koan.Cache.Adapter.Redis/`, want a patch release.

```pwsh
# 1. Commit with a conventional-commit subject. The prefix decides the bump magnitude.
git commit -m "fix(cache): Redis adapter stalls on slow Subscribe"

# 2. Preview what would happen on the next release.
pwsh scripts/versioning/Show-VersionStatus.ps1
#    Expect: Koan.Cache.Adapter.Redis: 0.8.0 → 0.8.1 (patch).
#    Everything else stays put.

# 3. Push your branch, open a PR to main, merge.
#    CI handles the rest — see nuget-publishing.md.
```

That's it. Nothing about the versioning system requires hand-editing.

### Kernel change

You modified an interface in `Koan.Cache.Abstractions`, `Koan.Data.Abstractions`, or any other kernel package. Every kernel package bumps together because they share a version.

Two paths, depending on how deliberate the change is:

**Auto path (default — what CI does on every merge):**

- A `feat:` or `feat!:` in a kernel folder will be picked up automatically.
- Kernel bumps minor (for `feat:`) or major (for `feat!:`).
- versions.props records `Kernel bump : minor (auto-detected from kernel-folder commits)`.
- No script invocation needed. Just commit with the right conventional-commit subject and merge.

**Manual path (for ADR-driven decisions where the audit trail matters):**

```pwsh
# After landing your kernel change with a `feat:` or `feat!:` commit:
pwsh scripts/versioning/Update-Versions.ps1 -BumpKernel minor -Reason "ARCH-0083 — Touch(timestamp) overload"
#    -BumpKernel: major | minor | patch
#    -Reason: required, gets recorded in versions.props header

# Review the delta:
git diff build/versions.props

# Commit and let CI tag/publish on merge.
git add build/versions.props
git commit -m "chore(release): bump kernel to 0.9.0 (ARCH-0083)"
```

**Pre-1.0 magnitude guidance:**

- `major` — kernel rewrites; rare
- `minor` — typical for ADR-driven additive contract changes
- `patch` — kernel-internal fix that doesn't change the contract

Post-1.0 the conventions tighten — `major` becomes "breaking change," `minor` becomes "additive only," `patch` becomes "bug fix only."

### Adding a package

You scaffolded a new csproj under `src/`. To wire it into the versioning system:

```xml
<!-- src/Koan.NewThing/Koan.NewThing.csproj -->
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <KoanPackageKind>Periphery</KoanPackageKind>    <!-- add this -->
    <TargetFramework>net10.0</TargetFramework>
    <Description>...</Description>                  <!-- required for NuGet -->
    <PackageTags>$(CommonPackageTags);...</PackageTags>   <!-- required -->
    ...
  </PropertyGroup>
</Project>
```

**Choosing kernel vs periphery:**

- Does the package export interfaces or value types other packages compile against? → **Kernel**. Add the relative path to [build/kernel-manifest.txt](../../build/kernel-manifest.txt). Adding to the kernel is an ADR-level decision; flag it for review.
- Anything else (implementation, adapter, connector, sample, service)? → **Periphery**. Just set `KoanPackageKind=Periphery`.

Then verify the metadata is complete:

```pwsh
pwsh scripts/versioning/Audit-NuGetMetadata.ps1
# Should report 0 missing Description / PackageTags / KoanPackageKind.
```

On the next release the package picks up the current kernel version (for kernel packages) or `$(KoanKernelVersion)` as its starting periphery version.

### Mark a package non-packable

Temporarily exclude a package from NuGet publishing — for example, a connector with unresolved compile errors that's being fixed on a different branch.

```xml
<!-- Inside the csproj's first <PropertyGroup>: -->
<IsPackable>false</IsPackable>
<!-- Add a brief comment explaining why and pointing at the tracking issue or branch. -->
```

Effect:

- The package no longer appears in the bumped-packages manifest.
- The release workflow ignores it.
- The csproj still builds (so other packages depending on it still work in-repo).
- versions.props still tracks it under its `KoanPackageKind=Periphery` (harmless; no NuGet emission).

To re-enable: delete the `IsPackable` line.

### Manual release

The release workflow handles releases on every merge to `main`. If you can't use CI (offline, on a fork, debugging the workflow), here's the manual flow:

```pwsh
# 1. Make sure main is up to date and your work is committed.
git switch main && git pull origin main

# 2. Compute versions. -AutoBumpKernel does what CI does.
pwsh scripts/versioning/Update-Versions.ps1 -AutoBumpKernel

# 3. Review the delta.
git diff build/versions.props
cat artifacts/bumped-packages.txt       # exact set of csprojs that will pack

# 4. Commit and tag.
git add build/versions.props
git commit -m "chore(release): bump versions"
pwsh scripts/versioning/New-Release.ps1 -Version <new-kernel-version> -Push

# 5. To publish, see nuget-publishing.md.
```

### Dry-run

You want to know what would happen without writing anything:

```pwsh
pwsh scripts/versioning/Update-Versions.ps1 -DryRun
# OR
pwsh scripts/versioning/Show-VersionStatus.ps1
# (Show-VersionStatus is just a wrapper that always runs in dry-run mode.)
```

Both print the bump table without touching versions.props.

To preview against a different reference tag (what-if):

```pwsh
pwsh scripts/versioning/Show-VersionStatus.ps1 -SinceTag release/v0.7.0
```

## Failure → recovery

### Symptom: `Show-VersionStatus.ps1` says "no bumps" but you expected one

**Why it happens:** the script compares against the most recent `release/v*` tag. If your commit landed on a feature branch but the tag is on `main`, and the branch's base is older than the tag, the diff might be empty.

**Recovery:**

```pwsh
# Verify the reference tag the script picked.
pwsh scripts/versioning/Show-VersionStatus.ps1
#   Look for "Reference tag : release/vX.Y.Z" in the output.

# Verify the commits the script SHOULD see for your package's folder.
git log <that-tag>..HEAD --oneline -- src/Koan.YourPackage/
#   If this is empty, your commit didn't touch that folder.
#   If it has entries, the script's range computation matched yours.
```

If `git log` shows commits but the script doesn't bump, check that the commit subjects match conventional-commit format. A commit with subject `update something` (no prefix) doesn't trigger a bump.

### Symptom: Wrong version computed (bumped major when you expected minor)

**Why it happens:** some commit in the folder's history since the reference tag has `!` after its type (e.g., `feat!:`) or `BREAKING CHANGE:` in the body. Auto-detect takes the highest magnitude across all commits.

**Recovery:**

```pwsh
# Find the offending commit.
git log <reference-tag>..HEAD --grep='!:' --grep='BREAKING CHANGE' -- src/Koan.YourPackage/
```

Options:

1. **Accept the major bump** if the change is genuinely breaking.
2. **Amend or revert** the commit subject if `!` was added by accident (only works if not yet merged).
3. **Override manually** for this release:
   ```pwsh
   pwsh scripts/versioning/Update-Versions.ps1 -BumpKernel patch -Reason "Override: <reason>"
   ```
   Only works for kernel bumps. For periphery, edit versions.props directly (rare; see anti-patterns).

### Symptom: versions.props doesn't match what's on main

For example, a workflow run created the bump commit and tag, but only the tag landed on origin (branch protection blocked the main push). main is at the pre-release commit; the tag points at a release commit that exists nowhere else.

**Why it happens:** the workflow's `git push origin main` was rejected by branch protection. The `git push origin <tag>` succeeded because tag pushes follow different rules.

**Recovery:**

```pwsh
# 1. Verify the inconsistency.
git fetch origin --tags --prune
git rev-parse origin/main
git rev-parse release/v<x.y.z>
#    If those differ AND release/v<x.y.z>'s commit isn't an ancestor of main, you have a dangling tag.

# 2. Delete the dangling tag — easier than trying to fast-forward main to a github-actions[bot] commit.
git push origin --delete release/v<x.y.z>
git tag -d release/v<x.y.z>

# 3. Fix the underlying cause (branch protection) — see nuget-publishing.md Failure → recovery.

# 4. Re-trigger the workflow by merging anything to main. Workflow will recompute and re-tag.
```

### Symptom: A new package doesn't appear in versions.props

**Why it happens:** the script discovers packable csprojs under `src/`. If your csproj isn't under `src/`, has `<IsPackable>false</IsPackable>` (directly or inherited from a parent `Directory.Build.props`), or has parse errors, it won't be picked up.

**Recovery:**

```pwsh
# 1. Confirm the csproj is in the audit list.
pwsh scripts/versioning/Audit-NuGetMetadata.ps1
#    Should show your package in the count. If not, IsPackable is suppressing it.

# 2. Check inherited IsPackable.
#    Look for Directory.Build.props files in parent folders (samples/, src/Services/, tests/ all have one).
#    Each sets IsPackable=false for everything under it.

# 3. If the package SHOULD be packable, either:
#    - Move the csproj out of the non-packable subtree.
#    - Or override at the csproj level: <IsPackable>true</IsPackable> in the csproj's first PropertyGroup.

# 4. Re-run audit and Update-Versions to confirm the package now appears.
pwsh scripts/versioning/Audit-NuGetMetadata.ps1
pwsh scripts/versioning/Update-Versions.ps1 -DryRun
```

### Symptom: versions.props edited by hand and now diffs are noisy

**Why it happens:** versions.props is generated. Hand-edits get overwritten on the next run, but they create temporary inconsistencies.

**Recovery:**

```pwsh
# Regenerate cleanly from the current state.
pwsh scripts/versioning/Update-Versions.ps1
git diff build/versions.props
#    If the diff is the cleanup of your hand-edit, commit.
#    If the diff also shows unwanted bumps, you have an unrelated issue — see other recoveries.
```

---

## Anti-patterns

- **Don't hand-edit `build/versions.props`.** It's generated by `Update-Versions.ps1`. Any hand-edit gets overwritten the next time the script runs. If you need to override a version, use `-BumpKernel` (for the kernel) or, as a last resort, hand-edit AND immediately commit AND skip the next auto-run (`[skip release]` in the merge commit).
- **Don't `git reset --hard` while versions.props or other generated files are uncommitted.** It silently destroys the work. If you need to discard local changes, use `git restore` on specific files, or commit your work to a throwaway branch first.
- **Don't bump the kernel for non-contract changes.** Internal refactoring of `Koan.Core` that doesn't change public types is a `chore:` (patch), not a `feat:` (minor). The kernel version is a contract claim — keep it honest.
- **Don't add packages to `build/kernel-manifest.txt` without an ADR.** That manifest defines the framework's contract surface. Adding to it is a framework-wide decision, not a casual edit.
- **Don't add `<Version>` directly to a csproj.** The version is resolved via `Directory.Build.targets` from versions.props. A direct `<Version>` element overrides the resolution and creates drift.

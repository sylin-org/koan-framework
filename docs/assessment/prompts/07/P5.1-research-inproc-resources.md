## Why this exists (the reframe)

The original P5.1 framed "sovereign / scales-down" as *strip the app to fit an AOT binary*. That was a
category error (caught in design review): it fused two independent axes —

- **Sovereignty** = *whose hardware / does data leave it* (your self-hosted box is fully sovereign).
- **Footprint** = *what runtime you need to deploy* (cloud → k8s → one Docker host → one native binary).

"Scales-down" is the **footprint** axis. A self-hosted Docker host with Postgres + Weaviate + Ollama is
already sovereign, scaled-down, **and full-featured** — and Koan already runs there (P5.2 demoed
self-orchestration of those containers on one machine). So that rung needs nothing.

The genuinely net-new rung is the **floor**: a single executable with **no container runtime at all** —
for appliances / edge / locked-down air-gapped boxes where Docker isn't an option. And the key insight:
**"single binary" must NOT mean "fewer capabilities" — it means every capability is satisfied by an
in-process resource instead of a server.** The work is therefore to **complete the in-process tier of the
capability ladder** by building adapters over *embedded* resources. This is not a framework-mechanism
question (the adapter seams already exist — ARCH-0084 capability tokens; the vector-repository interface
Weaviate implements; the AI embedding-generator interface the Ollama connector implements); it is a
**survey of the embeddable-resource ecosystem** and which we adopt.

## The in-proc tier today (what's covered, what's the gap)

| Pillar | In-proc resource shipping today | Status |
|---|---|---|
| Data (relational/doc) | **SQLite**, **JSON-file** (both in-proc, file-backed) | covered |
| Cache | **L1 memory** + **`Koan.Cache.Adapter.Sqlite`** (persistent in-proc) | covered |
| Jobs | **durable-on-sqlite ledger** | covered |
| Storage (blob) | **local filesystem** | covered |
| Web / MCP | in-proc by nature | covered |
| **Vector** | none — Milvus/Qdrant/Weaviate are all servers | **GAP** |
| **Embeddings** | none in-proc — only Ollama (separate process) | **GAP** |

Two primary gaps. (Verify this table empirically first — `src/Connectors/**`, the capability seams — it
may be stale.)

## Mission

Produce a **deep, evidence-based survey** of in-process / embeddable resources across the .NET (and
native-bindable) ecosystem that can be built as Koan adapters, then a **ranked decision brief**: which
adapters to build, in what order, under which posture. Be comprehensive — go past the two known gaps and
also surface richer in-proc options for the already-covered pillars (e.g. LiteDB / DuckDB / RocksDB for
data; an in-memory `System.Threading.Channels` bus for messaging) where they add real value.

### Candidate seed lists (start here, then go broader — do not treat as exhaustive)

- **Embedded vector** (ascending weight): pure-managed brute-force (`System.Numerics.Tensors` cosine over
  `float[]`, zero native deps, AOT-trivial, small/mid-N) · **sqlite-vec** (loadable SQLite extension —
  pairs with the existing SQLite data adapter so one file holds rows + vectors) · **USearch**
  (`cloud.unum.usearch` .NET bindings, HNSW) · **hnswlib** · LanceDB(.NET maturity?) · Faiss(.NET) ·
  DiskANN · SK's `Microsoft.SemanticKernel.Connectors.SqliteVec` (prior art to study, not necessarily adopt).
- **In-proc embeddings** (the model runtime is the fork): **ONNX Runtime** (`Microsoft.ML.OnnxRuntime`) +
  a small model (all-MiniLM / bge-small ONNX) + **`Microsoft.ML.Tokenizers`** (BERT/WordPiece) — the
  canonical .NET path, single process · **LLamaSharp** (llama.cpp / GGUF — embeddings *and* chat in-proc) ·
  Microsoft.Extensions.AI local providers · note: **no fast pure-managed transformer inference exists**, so
  truly-in-proc embeddings REQUIRE a native runtime; the alternative is a co-located Ollama process (still
  no cloud, no Docker, just two local processes).
- **Data (richer in-proc)**: LiteDB (pure-managed NoSQL doc) · DuckDB.NET (embedded OLAP, native) ·
  RocksDB/LevelDB (embedded KV, native) · evaluate whether any beats the sqlite/json baseline for a real use.
- **Messaging (in-proc)**: a `System.Threading.Channels`-backed in-process bus — does the messaging pillar
  already have one? (the jobs ledger is the durable substrate; in-proc pub/sub may be a small gap).

### Evaluation rubric (score every serious candidate)

For each: **maturity / activity / .NET-first-class?** · **license** (must be permissive — Apache/MIT/BSD;
flag GPL/AGPL) · **native vs pure-managed** (RIDs supported; linux-x64/arm64/win-x64) · **NativeAOT +
trimming friendliness** (the floor rung wants AOT — this is decisive) · **the Koan seam it implements**
(which interface + which ARCH-0084 capability tokens it can honestly declare; what it must degrade/fail-loud
on) · **effort to adapt** (S/M/L) · **the one trade-off that matters**. Use real evidence (package pages,
repos, AOT-compat notes) — and **adversarially verify** AOT/license/maintenance claims (they rot).

### The two posture decisions to RESOLVE

1. **Floor posture — pure-managed or native-allowed?** A *zero-native* floor (sqlite + brute-force-managed
   vector + memory cache, embeddings external via co-located Ollama) is maximally AOT-clean and portable. A
   *native-allowed* floor (+ ONNX-RT embeddings + sqlite-vec) yields a *truly self-contained* full-featured
   single binary. Recommend one (or a two-rung ladder: managed-floor + native-floor).
2. **Sequencing.** Likely cheapest-first: the pure-managed brute-force vector adapter (small, AOT-clean,
   immediately makes semantic search work in a single managed binary against external embeddings), then the
   ONNX in-proc embedding adapter (the "no separate process at all" payoff). Confirm or revise from evidence.

## Output (this card's deliverable)

A decision brief at `docs/assessment/evidence/inproc-adapter-survey.md` (or an ADR-candidate under
`docs/decisions/`): the scored candidate matrix, a **recommended in-proc adapter set** (per pillar, ranked),
the resolved floor posture, the build sequence, and for each recommended adapter a one-paragraph "adapter
shape" sketch against the real Koan seam (the interface + capability tokens it implements). **No production
adapters built in this card** — it ends in the brief + the follow-on impl cards it spawns (e.g.
`X-inproc-vector-bruteforce`, `X-inproc-embeddings-onnx`, then the AOT-substrate + `S2.Sovereign` proof).

## How to run it

Good fit for a **multi-agent research workflow** (Ultracode): parallel readers — one per pillar/gap (vector,
embeddings, data, messaging) + one on AOT/trim reality + one on the existing seams — returning structured
scored candidates, then an **adversarial-verification** pass on the AOT/license/maintenance claims, then the
orchestrator synthesizes the ranked brief. Keep findings empirical (real package/repo evidence), and treat
every "AOT-compatible" claim as guilty until verified.

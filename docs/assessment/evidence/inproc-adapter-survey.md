# In-process resource survey — completing the capability ladder's in-proc tier (P5.1)

## 0. Floor RID set (declared once)

The **floor RID set** for every scoring decision below is **`linux-x64`, `linux-arm64`, `win-x64`**. `osx-*` and `win-arm64` are *not* floor targets — they are noted where a candidate's coverage diverges, but their absence is never disqualifying and their presence is never load-bearing. Every RID claim in §4/§5/§8 is evaluated against exactly these three.

## 1. Summary & the reframe

**Sovereignty and footprint are orthogonal axes, and conflating them has muddied P5.1.** Sovereignty is *whose hardware* runs the capability (your box vs a cloud tenant). Footprint is *what runtime shape* the capability ships in (a co-located process / container vs an in-process resource linked into the app binary). Koan already serves sovereignty well: every pillar has a self-hostable provider you can run on your own metal. What it does **not** yet have at every rung is the tightest footprint: **a single binary, no container runtime, where every declared capability is satisfied by an in-process resource.**

The reframe that matters: **"single binary" is not "fewer capabilities."** The goal is the opposite — every capability available *without reaching out of process*, not a stripped subset. The sweep confirms the in-proc *coverage* floor is already broad (data/cache/jobs/storage/web all have honest in-proc resources). That is a **footprint** statement, not an AOT-readiness statement — see §3. Exactly **two capability classes have a genuine, unfilled in-proc gap**:

1. **Embedded vector search** — no shipping `IVectorSearchRepository` runs in-process; every impl targets an external server/container (Weaviate, Qdrant, Milvus, ES/OS).
2. **In-proc embeddings** — every `IEmbedAdapter` is an `HttpClient` to an external model server (Ollama, LMStudio, HuggingFace, ZenGarden).

A third item surfaced as a **correction, not a target**: messaging's "open question" resolves to a **GAP** — no in-proc `System.Threading.Channels` bus ships; the only transport is the RabbitMQ broker, and the boot report's advertised `InMemoryProvider.Available=true` / reserved `InMemory` Priority=10 slot points at a **provider class that does not exist**. That self-report overstates capability and should be fixed alongside the vector/embeddings work.

This brief scores candidate in-proc resources for the two real gaps (plus the messaging GAP fix), grounds each against the **real adapter seam**, and resolves two posture decisions: the floor's runtime posture (managed vs native vs two-rung) and the build sequence.

## 3. AOT reality verdict

**Can a managed-only floor publish NativeAOT today? Yes — and the harder version of it already does, for one specific assembled floor.** Exactly one project AOT-publishes (`samples/guides/g1c1.GardenCoop`), via `-p:PublishAot=true` on the command line (`start-native.bat:35`), not in the csproj. It is a real ilc-compiled native binary (verified 2025-11-12: ~22-28 MB win-x64, symbols stripped, SQLite embedded via SQLitePCLRaw self-contained, dashboard + automation + graceful shutdown all working). The plain `dotnet run` path stays IL/JIT.

**The critical honesty boundary:** what is *proven* AOT-publishable is **GardenCoop's actual floor** — sqlite + `Entity<T>` + Newtonsoft + Web + admin/auth. The Rung-A delta proposed in §6 (brute-force managed vector + memory cache + Channels messaging) is **argued lower-risk, not spiked**. The brute-force vector scan is plain managed SIMD compute (no native interop, no reflection-heavy serialization) — *structurally* lower AOT risk than what already ships; `System.Numerics.Tensors` is in-box dotnet/runtime, built and tested under NativeAOT in runtime CI. But "the *assembled* Rung-A floor AOT-publishes" is an inference from proven parts, never a measured fact. It needs a spike (`S2.Sovereign-proof`) before it is asserted.

Critically, **GardenCoop's floor is sqlite + `Entity<T>` + the full Newtonsoft hot path, and it AOT-runs with it live.** `SqliteRepository.cs:419/425` does `JsonConvert.DeserializeObject`/`SerializeObject` on every read/write. So **Newtonsoft does not hard-block NativeAOT on .NET 10 — it warns.** Under AOT the trimmer can't statically prove which serialized members are reachable, so it emits trim/AOT warnings and *can* fault at runtime if a serialized type was trimmed — resolved by **rooting** (`NativeAotRoots.xml` `TrimmerRootDescriptor` + `[DynamicDependency]`), which is exactly GardenCoop's mitigation. **The SQLite leg additionally hard-requires `--self-contained true`** at publish (else `DllNotFoundException: e_sqlite3`) plus those trim roots — AOT here is "works, with curation," never "works, free."

**The Newtonsoft tension is to MANAGE, not relitigate.** Per the assessment (§6.1), Newtonsoft is the canonical app-facing serializer; STJ source-gen stays adapter-internal for wire formats. It is *not* isolatable as "remove Newtonsoft from the floor" — it is structurally on the SQLite/Entity hot path. It *is* isolatable as "root the entity types the serializer touches," which is what the descriptor does.

**The genuine engineering debt (blockers? no. taxes? yes):**
- **The reflective assembly-closure scan is the single most AOT-hostile seam, and it's on the floor.** `AppBootstrapper.InitializeModules` (`AppBootstrapper.cs:83`) does `AppDomain.CurrentDomain.GetAssemblies()` + a `GetReferencedAssemblies`/`Assembly.Load` fixpoint + a `Directory.GetFiles("Koan.*.dll")` base-dir scan. AOT tolerates this via force-rooting + lenient-skip (`catch { lenientAssemblySkips++ }`), so it degrades gracefully rather than blocking — but the discovery model is reflective-by-construction.
- **The "AOT-friendly source-gen discovery" claim is half-true.** The registry generator *does* emit a build-time manifest (AOT-correct), but `AppBootstrapper` *reaches* it reflectively (`Assembly.GetType("…RegistryManifestLoader")` + `GetMethod` + `Invoke`; `Activator.CreateInstance(initializerType)`). It works under AOT (GardenCoop proves it) but is not the clean fully-static path the framing implies.
- **Trim-root maintenance** is the ongoing tax: every reflection-touched/serialized type must be rooted as you add modules.

**Bottom line:** A managed-only floor (sqlite + managed brute-force vector + memory cache + Channels messaging; embeddings external via co-located Ollama) can publish NativeAOT today — there is **no hard blocker**, only curation that is already built and exercised *for GardenCoop's floor*. The added managed primitives are lower-risk-but-unspiked. Adding native deps (ONNX-RT, sqlite-vec) is where AOT confidence drops to **unverified** (see §6).

## 5. Recommended in-proc adapter set, ranked

**1. Brute-force cosine vector adapter (`System.Numerics.Tensors`) — ADOPT FIRST.**
Implements `IVectorSearchRepository<TEntity,TKey>`, produced by an `IVectorAdapterFactory : INamingProvider` decorated `[ProviderPriority(N)]` + `[KoanService(ServiceKind.Vector)]` and registered by the adapter's `KoanAutoRegistrar.Initialize → AddSingleton<IVectorAdapterFactory, X>()`. Storage is a per-partition `ConcurrentDictionary<TKey, float[]>` (optionally persisted to the shipped sqlite/json data adapter via `Entity<T>`); `Search(VectorQueryOptions)` runs `TensorPrimitives.CosineSimilarity` across the set and partial-sorts to `TopK`, returning `VectorQueryResult<TKey>(Matches…)`. **Honestly declares** (via `IDescribesCapabilities.Describe(ICapabilities)`) `VectorCaps.Knn`, `BulkUpsert`, `BulkDelete`, `ScoreNormalization` (cosine ∈ [-1,1]), `DynamicCollections` (per-partition dicts are free), and `Filters` with `FilterSupport.Uniform` (metadata predicate evaluated in managed code post-scan — fully correct, not pushed-down). **Fails loud / does not declare** `Hybrid` (no BM25), `NativeContinuation`, `StreamingResults` (offset paging only), `AtomicBatch`, `MultiVectorPerEntity`. **Degrades on scale**: O(N) per query — honest for ≤ ~10⁵ vectors, not millions. AOT-trivial, every floor RID, zero infra. This is also the **correctness oracle** for every other vector adapter.

**2. ONNX-RT in-proc embeddings adapter — ADOPT (native rung).**
Implements `IEmbedAdapter.Embed(AiEmbeddingsRequest) → AiEmbeddingsResponse{Vectors, Model, Dimension}` over `IAiAdapter`. The AI seam is structurally different: **no factory, no `[ProviderPriority]`** — registered by an `IAiAdapterContributor` (added via `KoanAutoRegistrar.Initialize → TryAddEnumerable<IAiAdapterContributor>(…)`) whose `Contribute` calls `IAiAdapterRegistry.Add(adapter)`, mirroring `OllamaAdapterContributor`. Runs `all-MiniLM-L6-v2` (Apache-2.0, 384-dim) on `Microsoft.ML.OnnxRuntime`, tokenized by the **AOT-clean `FastBertTokenizer`** (lead with it, not `Microsoft.ML.Tokenizers.BertTokenizer` whose type inventory the skeptic could not reconfirm). Declares `AiCapability.Embed` **only** (structural — implements only `IEmbedAdapter`) — fails loud on `Chat`/`Vision`/`Streaming`/`Tools`, on missing/incompatible ONNX model or vocab, and reports `Model`+`Dimension` truthfully so `EmbeddingWorker`'s `VectorModelGuard.GuardWrite` mixed-space guard works. Closes the embeddings GAP for a true single binary. **Caveat (load-bearing):** ORT is native and **NativeAOT-unverified** — it converts the floor from pure-managed to native-allowed.

**3. sqlite-vec vector adapter — ADOPT (native rung 2, full single binary).**
Same composition trio as adapter 1 — `IVectorAdapterFactory` (`[ProviderPriority(N)]` + `[KoanService(ServiceKind.Vector)]`, registered via `KoanAutoRegistrar.Initialize → AddSingleton<IVectorAdapterFactory, X>()`) producing an `IVectorSearchRepository<TEntity,TKey>` over the **same `SqliteConnection`** the shipped `Koan.Data.Connector.Sqlite` uses — one `.db` file holds rows *and* vectors in a `vec_*` virtual table; `Upsert`/`Delete` = `INSERT`/`DELETE`, `Search` = `… WHERE embedding MATCH ? ORDER BY distance LIMIT k`. Declares `VectorCaps.AtomicBatch` (real SQL txn) and `MultiVectorPerEntity` over the brute-force set; `Filters` constrained to **equality** (mirror SK's envelope); **NOT** `Hybrid`. **Two corrections must land in the adapter:** (a) the .NET binding bundles a May-2025 `vec0` predating the v0.1.8/v0.1.9 **DELETE bug fixes** — pin/ship a newer native `vec0` since the adapter does DELETE; (b) under SDK `PackageReference` the package copies nothing, and `LoadExtension` bypasses .NET RID resolution — **the adapter must deploy + resolve the `runtimes/{rid}/native/vec0` path itself before load**. AOT confidence is **uncertain** (runtime native dynamic-load is fragile under fully-static NativeAOT) — gate on a spike.

**4. InProcess Channels messaging provider — ADOPT (GAP fix).**
Implements `IMessagingProvider` at Priority=10, backed by `Dictionary<Type, Channel<object>>` + a pump task, honoring the `IMessageConsumer` `Pause`/`Resume`/`IsActive` contract. Non-durable by design (durable work leans on the Jobs sqlite ledger). Fills the **reserved-but-empty** slot and lets the boot report stop lying. Substrate is AOT-clean; the residual AOT liability is the shared messaging pipeline's reflection, which RabbitMQ already carries.

**Deferred / conditional:** HNSW.Net or HnswLite as a managed-ANN scale rung (only after a `PublishAot` persistence spike); USearch for x64/osx-arm64 scale hosts (never the arm64 floor); DuckDB.NET for an embedded-OLAP *data* adapter or as the spike target if a durable-native vector store is wanted despite the experimental-persistence blocker.

## 6. RESOLVE — Posture decision 1: floor posture

**Decision: ship a two-rung ladder. The default floor is pure-managed; a native-allowed rung is the opt-in full single binary.**

- **Rung A — pure-managed floor (default):** sqlite + brute-force-managed vector (`TensorPrimitives`) + memory cache + in-proc Channels messaging; **embeddings external via co-located Ollama**. **AOT status, stated honestly:** the *managed primitives are AOT-clean by construction* (Tensors is in-box dotnet/runtime; Channels is on no AOT-limitations list; memory cache is managed), and *GardenCoop's actual floor* (sqlite + Entity + Newtonsoft + Web) is *proven* AOT-publishable — but **the assembled Rung-A floor is unspiked**: "brute-force vector + Channels + memory cache + sqlite together AOT-publish" is an inference from proven parts, not a measured result. **It additionally inherits the sqlite leg's hard requirements: `--self-contained true` + `NativeAotRoots.xml` trim roots** (else `DllNotFoundException: e_sqlite3` and/or runtime trims). Semantic search works in a single managed binary against externally-produced embeddings; the embedding *step* is the one capability that legitimately reaches out of process (to a second local process, no cloud, no Docker).

- **Rung B — native-allowed floor (opt-in, truly single binary):** Rung A + ONNX-RT embeddings + sqlite-vec. This is the only configuration where **both embed and search happen in-process with zero external process** — a genuine single self-contained binary. It costs the pure-managed AOT guarantee: ORT and sqlite-vec are native and NativeAOT-**unverified** (every "AOT-compatible" claim for the native candidates is flagged, not upstream-confirmed).

**Rationale.** A single posture would force a false choice. Collapsing to pure-managed forecloses the "one binary, zero processes" story that is the whole point of the rung. Collapsing to native-allowed forfeits the clean-NativeAOT floor that already works and abandons sovereignty-conscious AOT deployments. The two rungs are the same code paths behind Reference=Intent: reference the managed vector adapter → Rung A; add the ONNX-RT + sqlite-vec packages → **Rung B self-elects by Reference=Intent** (no config switch — the package references *are* the selection). The brute-force adapter doubles as the correctness oracle for sqlite-vec, so Rung A is never throwaway. **This is the no-stopgap answer:** neither rung drops a capability to a floor — both deliver the full capability, differing only in footprint.

## 7. RESOLVE — Posture decision 2: build sequencing

**Decision: (1) brute-force managed vector → (2) ONNX-RT embeddings → (3) sqlite-vec → (4) Channels messaging. Vector before embeddings.**

Justification from the evidence:

1. **Brute-force vector first** — effort **S**, AOT-trivial and CONFIRMED, zero new dependencies, and it is the **correctness oracle** every other vector adapter is validated against (so it must exist before sqlite-vec can be trusted). It immediately makes semantic search work in a managed binary against *external* embeddings, delivering value before the embeddings axis is touched. Lowest risk, highest leverage.

2. **ONNX-RT embeddings second** — the `EmbeddingWorker` bridge is `embed → GuardWrite → vector Upsert`; once the vector seam exists (step 1), wiring an in-proc embedder closes the loop end-to-end. The survey itself sequences it as "the second build, after the vector adapter." It is effort **M** and the first native dependency — doing it second means the native-AOT risk is introduced *after* a proven managed baseline exists to fall back to.

3. **sqlite-vec third** — effort **M**, but gated on the brute-force oracle (step 1) for recall validation and on the binding/native-path corrections from §5. It is the durable, single-file upgrade over the in-memory brute-force scan.

4. **Channels messaging last** — independent of the vector/embeddings axis, effort **S→M**; it can in fact run in parallel at any point, but it is the lowest-urgency GAP (the boot-report honesty fix is the real driver).

## 9. Open questions / risks

- **The assembled Rung-A floor is unspiked.** Only GardenCoop's *actual* floor (sqlite + Entity + Newtonsoft + Web) is proven AOT-publishable. "Brute-force vector + Channels messaging + memory cache + sqlite together AOT-publish" is *inferred from AOT-clean managed primitives*, never measured. `S2.Sovereign-proof` (target `linux-arm64`) is the spike that closes this — and the SQLite leg's `--self-contained true` + `NativeAotRoots.xml` requirements must be carried explicitly into Rung A's publish recipe.
- **ORT NativeAOT is unverified, not proven-broken.** The cited failures (#18363/#21668) are iOS *Mono* "aot-only mode," a different runtime from desktop/server ILC; no positive proof that `InferenceSession` works under `PublishAot` on linux-x64/win-x64 was found either. Honest framing: **"NativeAOT-clean is unverified (leaning broken)"** — Rung B's AOT claim needs a real spike before it's asserted. Same posture for sqlite-vec's `LoadExtension` under fully-static AOT.
- **sqlite-vec binding staleness is a concrete correctness risk, not cosmetic.** The bundled May-2025 `vec0` predates the v0.1.8/v0.1.9 DELETE fixes and the adapter does DELETE. The native binary must be refreshed independently of the year-stale .NET wrapper, or the adapter ships a known DELETE bug.
- **VSS (DuckDB) durable-vector persistence is unsafe today** — experimental, documented data-loss/corruption, full reserialize per checkpoint. If a durable *native* in-proc vector store is ever wanted, this blocks DuckDB until upstream stabilizes; sqlite-vec is the safer native path meanwhile.
- **`FilterSupport.Full` AND `Hybrid` for a SQL-backed vector adapter (sqlite-vec/DuckDB) are design assertions, not free tokens.** The seam's current default is `FilterSupport.Uniform` ("vector metadata is schemaless"); real column pushdown *and* the SQL-FTS+vector `Hybrid` blend are both adapter work, not inherited. Declare the *honest* constrained envelope (equality for sqlite-vec, no `Hybrid`) until the work is done.
- **The reflective boot scan is the structural AOT tax across both rungs.** It tolerates AOT via force-rooting + lenient-skip but does not natively love it; the "AOT-friendly source-gen discovery" framing is half-true (manifest is generated, but reached reflectively). `X-aot-substrate` should make the curation surface a first-class, documented thing rather than per-sample tribal knowledge.
- **Newtonsoft is a managed tension to maintain, not a defect to fix.** It is canonical (assessment §6.1) and the floor demonstrably AOT-runs with it via rooting. The co-located-Ollama connector additionally pins Newtonsoft 13.0.4 → IL2026 warnings; migrating *that connector* to STJ source-gen is the only place a warning-free managed embeddings fallback would require work — optional, scoped to the connector, not the floor.
- **Messaging AOT liability is the shared pipeline, not the provider.** `MakeGenericMethod` + Expression trees + reflection-STJ in `MessagingExtensions`/`IMessageBuffer` are inherited by *any* provider; the Channels provider is no worse than RabbitMQ. If a warning-free messaging floor is ever required, that pipeline — not the provider choice — is the target; **`X-aot-substrate` now owns it** (previously raised in §9 with no owner).
- **SQLite cache/data leg has a deprecated package ID.** The pinned `SQLitePCLRaw.lib.e_sqlite3` is NuGet-deprecated as a rename to `SourceGear.sqlite3` but still publishes current SQLite (3.50.3 verified live) — cosmetic migration debt on the floor's SQLite leg, not a blocker; fold the ID migration into whichever card touches the sqlite adapters.
- **Thinly-verified items, stated plainly:** HNSW.Net/HnswLite AOT persistence (needs a `PublishAot` spike — not assumed clean); `Microsoft.ML.Tokenizers.BertTokenizer` type inventory (reconfirm or default to FastBertTokenizer); DuckDB.NET consumer-side AOT (IL internals are AOT-shaped but the ADO.NET layer carries no trim/AOT annotation). Treat every native candidate's "AOT-compatible" as unverified until a spike says otherwise; only the pure-managed candidates (Tensors, and managed-core HNSW) are AOT-safe by construction.

# E08 — Certmesh CSR Enrollment (Keys Never Travel)

**Repo(s)**: Koi (ZG follows in E09) · **Phase**: C · **Prereqs**: E05 (token story — this
rides the same auth surface), E03 (republish after this lands) · **One session**
Read [CHARTER.md](CHARTER.md) in full first.

## Mission

Fix the single biggest PKI-correctness defect in the stack: certmesh has **no CSR
mechanism** — the CA generates member private keys server-side and ships them over the wire
on enrollment and renewal. Keys that traveled are not proof-of-possession anchors, which
caps everything the trust column can ever claim (STACK-0001 item 6 names this a
prerequisite for mTLS-grade workload identity). Convert enrollment and renewal to CSR-based
issuance: the private key is generated by the enrollee and never leaves it.

## Context (verify each)

- Current behavior: `JoinResponse.service_key` and `RenewRequest.key_pem` carry private
  keys over the wire (`KOI/docs/assessment/findings/reader-certmesh-deep.md:31,80`; confirm
  in `KOI/crates/koi-certmesh/src/protocol.rs` or equivalent).
- The enrollment surface: `POST /v1/certmesh/join` (TOTP-authenticated), `POST
  /v1/certmesh/renew`; route table `KOI/crates/koi-certmesh/src/http.rs:40-107`.
- Consumers who must keep working (they update in E09): ZG's pond
  (`ZEN/src/moss/src/api/v1/pond.rs:454-489` builds `JoinRequest`; rake's enrollment at
  `ZEN/src/rake/src/enrollment.rs`).
- Crypto stack: certmesh uses ECDSA P-256 (verify in koi-crypto); rcgen (or whatever the
  current cert dependency is) supports CSR (`CertificateSigningRequest`) — verify the
  dependency's actual API before designing.
- Frozen constants (CHARTER rule 3) are nearby in koi-crypto — do not touch them.

## DECIDED

1. **Protocol v2**: `JoinRequest` carries a PKCS#10 CSR (PEM); the CA validates
   (key type/curve, requested CN/SANs against policy), signs, returns `ca_cert` +
   `service_cert` — **no private key fields in any response**. `Renew` likewise carries a
   CSR (fresh key or rekey, enrollee's choice).
2. **Replace, don't bridge** (greenfield posture + pre-launch reality): the server-side
   keygen path is deleted, not feature-flagged. Existing enrolled meshes re-enroll —
   document the one-page re-enroll procedure. (The operator accepted breaking changes of
   this class in the Epic analysis; the TOTP ceremony itself does not change.)
3. The CLI (`koi` enrollment commands) and koi-embedded grow the keygen+CSR client side, so
   every first-party consumer does the right thing by default.
4. An ADR in Koi's `docs/adr/` records the change (cite the defect, the PoP rationale, the
   re-enroll consequence), cross-referencing STACK-0001 item 6.
5. **Guards**: integration tests — enroll via CSR end to end (key generated client-side,
   cert chains to CA, key never appears in any captured response); renewal; policy
   rejection (wrong curve / disallowed SAN).

## DEFAULT

- Rekey-on-renew as the default client behavior (fresh key each renewal); keep-key as an
  option.
- Version the protocol structs explicitly (the wire format is a contract — E07's certmesh
  fixtures must be updated in the same session).

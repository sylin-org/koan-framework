# Versioning workbook

Operational guide for the ARCH-0082 two-tier versioning system. **Read this when you need to ship a release or you're not sure what version a change should produce.**

Companion ADR: [ARCH-0082](../decisions/ARCH-0082-versioning-strategy.md).

---

## Mental model in 30 seconds

- **Kernel packages** (14 total — see [build/kernel-manifest.txt](../../build/kernel-manifest.txt)) share one version: `$(KoanKernelVersion)`. They move together.
- **Periphery packages** (everything else under `src/`) have independent versions. Each bumps only when its own folder has commits since the last release tag.
- One file is the source of truth: [build/versions.props](../../build/versions.props). Generated by tooling; committed to git.
- One tag marks each release: `release/v<kernel-version>` (e.g., `release/v0.7.1`).
- **Default cycle: zero-touch.** Merge `dev → main` and the [release-on-main workflow](../../.github/workflows/release-on-main.yml) computes versions, commits the bump, tags, and (optionally) publishes to NuGet.

---

## The happy path (zero-touch cycle)

This is what 95% of releases look like for the standard `dev → main` workflow:

```
1. Work on dev
   ↳ Commit with conventional-commit subjects (feat: / fix: / chore: / ...).
     The script uses these subjects to decide bump magnitude.

2. Open a PR dev → main, merge it.

3. Wait ~2 minutes.
   ↳ The release-on-main workflow runs:
       a. Builds Koan.sln (refuses to release if broken)
       b. Runs Update-Versions.ps1 -AutoBumpKernel
       c. If versions.props changed:
            - commits "chore(release): bump versions [skip release]"
            - tags release/v<kernel-version>
            - pushes both
       d. If KOAN_NUGET_PUBLISH=true AND NUGET_API_KEY is set:
            - dotnet pack Koan.sln
            - dotnet nuget push *.nupkg

4. Delete dev locally + on origin, branch new dev from main.

5. Repeat.
```

There is no step 4.5 "remember to update versions." There's no step 6 "tag the release." The workflow handles both.

**What if a merge has no version-worthy changes?** (docs-only, tests-only, etc.) The workflow detects no bumps, skips the commit/tag/publish entirely, and reports "Nothing to release" in the run summary.

**Opting out of a specific release:** include `[skip release]` in the merge commit subject. The workflow short-circuits at the job-level conditional.

### Enabling NuGet auto-publish (optional)

Off by default. To turn it on:

1. **Repository variable** (Settings → Secrets and variables → Actions → Variables):
   - Name: `KOAN_NUGET_PUBLISH`
   - Value: `true`

2. **Repository secret** (Settings → Secrets and variables → Actions → Secrets):
   - Name: `NUGET_API_KEY`
   - Value: your nuget.org API key

The workflow only publishes when BOTH are set. Missing either → packages get computed and tagged on main, but nothing is pushed to nuget.org.

### What the workflow guarantees

- **Loop-safe.** Its own commits use `GITHUB_TOKEN`; GitHub Actions deliberately does not retrigger workflows for `GITHUB_TOKEN` pushes.
- **Build-gated.** A red build aborts the release before any tag is created.
- **Idempotent.** Re-running on a commit whose tag already exists logs a warning and skips. No clobbered tags.
- **Serialized.** The `concurrency: release-main` group means two near-simultaneous merges process in order, never in parallel.

## Files you'll touch (or not)

| File | Hand-edit? | Notes |
|---|---|---|
| `build/versions.props` | **No** — regenerated by `Update-Versions.ps1` | Reviewable diff; commit alongside release. |
| `build/kernel-manifest.txt` | **Rarely** — ADR-level decision | Adding/removing kernel packages changes the framework contract surface. |
| Each csproj's `<KoanPackageKind>` | **No after migration** | Set once during the ARCH-0082 migration. New packages set it manually (see "Adding a new package"). |
| `Directory.Build.targets` | **No** | Holds the resolution logic that maps `KoanPackageKind` → `<Version>`. |

## Scenario 1 — Manual periphery bump (when CI is unavailable)

> **You usually don't need this.** Merge to `main` and the workflow handles bumps. Use this path when you're releasing locally (no GitHub Actions), reproducing a release for a hotfix branch, or you want to preview the exact commit/tag the workflow would produce.

```pwsh
# 1. Commit your work as usual with a conventional-commit subject.
git commit -m "fix(cache): Redis adapter stalls on slow Subscribe"

# 2. Preview what would change on the next release.
pwsh scripts/versioning/Show-VersionStatus.ps1

#    Look for "Koan.Cache.Adapter.Redis" in the bumps table; expect a patch bump.
#    Other periphery packages without commits since the last release stay put.

# 3. Generate the new versions.props.
pwsh scripts/versioning/Update-Versions.ps1

# 4. Review and commit.
git diff build/versions.props        # one or two version lines change
git add build/versions.props
git commit -m "chore(release): bump Koan.Cache.Adapter.Redis to 0.7.1"

# 5. Tag and (optionally) push.
pwsh scripts/versioning/New-Release.ps1 -Version 0.7.0 -Push
#    -Version is the KERNEL version (which didn't change here). The tag marks
#    the commit; per-package versions are recorded in build/versions.props at
#    that commit.
```

Why no kernel bump? Because nothing in the kernel folders changed. The kernel only moves when you say so (Scenario 2).

## Scenario 2 — Kernel bump for an ADR

Two paths, depending on whether you trust the auto-detection or want an explicit audit trail.

**Auto path (default for the merge-to-main workflow):**
- Your conventional-commit subjects ARE the trail.
- A `feat!:` or `feat:` in a kernel folder will be picked up by `-AutoBumpKernel` and bumps the kernel accordingly.
- versions.props records `Kernel bump : minor (auto-detected from kernel-folder commits)` in its header.
- No manual script invocation needed.

**Manual path (for high-ceremony ADR-driven bumps, or local dry-runs):**
The contract surface changed — e.g., a new method on `ICacheStore`, a property renamed in `Koan.Data.Abstractions`. Every kernel package bumps together. Periphery packages with changes in their own folders also bump (independently).

```pwsh
# 1. Land your kernel change with conventional commits as usual.
git commit -m "feat(cache)!: add Touch(timestamp) overload to ICacheStore (ARCH-0083)"

# 2. Bump the kernel deliberately. -Reason is required and gets recorded.
pwsh scripts/versioning/Update-Versions.ps1 -BumpKernel minor -Reason "ARCH-0083 — Touch(timestamp) overload"

#    -BumpKernel options:
#      major  →  0.7.0 → 1.0.0
#      minor  →  0.7.0 → 0.8.0    (typical for additive contract changes)
#      patch  →  0.7.0 → 0.7.1    (rare; only for kernel-internal fixes)

# 3. Review the delta. KoanKernelVersion changes; periphery packages with
#    folder commits also bump.
git diff build/versions.props

# 4. Commit + tag.
git add build/versions.props
git commit -m "chore(release): bump kernel to 0.8.0 (ARCH-0083)"
pwsh scripts/versioning/New-Release.ps1 -Version 0.8.0 -Push
```

**Magnitude guidance** (pre-1.0):

- `major` — really substantial; reserved for kernel rewrites
- `minor` — typical for ADR-driven additive contract changes
- `patch` — kernel-internal fix that doesn't change the contract

Post-1.0 the conventions tighten — `major` becomes "breaking change," `minor` becomes "additive," `patch` becomes "bug fix only."

## Scenario 3 — I just want to see what would happen on release

Read-only, no side effects.

```pwsh
pwsh scripts/versioning/Show-VersionStatus.ps1
```

Output sections:
- Reference tag (last `release/v*` tag, or "no prior release tag")
- Kernel: current version, "held" message (no `-BumpKernel` flag)
- Periphery bumps: table of packages that would bump, with current/new versions and commit counts
- Either "Periphery: no bumps …" or the bumps table

What-if a different reference tag:
```pwsh
pwsh scripts/versioning/Show-VersionStatus.ps1 -SinceTag release/v0.6.4
```

## Scenario 4 — Adding a new package to the framework

After scaffolding a new csproj under `src/`, you need to tell the versioning system about it.

```xml
<!-- src/Koan.NewThing/Koan.NewThing.csproj -->
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <KoanPackageKind>Periphery</KoanPackageKind>    <!-- add this -->
    <TargetFramework>net10.0</TargetFramework>
    ...
  </PropertyGroup>
</Project>
```

**Choosing kernel vs. periphery**:
- Pure interfaces + value types that other packages compile against? → **Kernel**. Add the path to `build/kernel-manifest.txt`. Bumping kernel = ADR-level decision.
- Anything else (implementation, adapter, connector, web extension, sample, service)? → **Periphery**.

Then run `Update-Versions.ps1` once to add the new package to `build/versions.props`:

```pwsh
pwsh scripts/versioning/Update-Versions.ps1
```

Or manually edit `versions.props` if you prefer (the tool is idempotent — re-running converges).

## Scenario 5 — Publishing to NuGet

The `New-Release.ps1` script can run the full `dotnet pack` + `dotnet nuget push` flow.

```pwsh
$env:NUGET_API_KEY = "oy2..."
pwsh scripts/versioning/New-Release.ps1 -Version 0.7.1 -Push -PublishNuGet
```

Preconditions the script enforces:
- Working tree must be clean (no staged or modified files)
- `build/versions.props` must exist and its `KoanKernelVersion` must match `-Version`
- No existing `release/v<Version>` tag
- (informational) HEAD on `main`, `dev`, `feat/*`, or `release/*` — warns otherwise

For a dress rehearsal:
```pwsh
pwsh scripts/versioning/New-Release.ps1 -Version 0.7.1 -DryRun
```

Verifies all preconditions and prints the plan.

## Scenario 6 — A periphery package was unintentionally bumped

The tool said `Koan.Foo` bumped, but you don't think it should have. Causes:

1. **A commit touched the folder that you didn't expect.** Run:
   ```pwsh
   git log <last-release-tag>..HEAD -- src/Koan.Foo/
   ```
   That's the exact range the tool inspects.

2. **A wide commit subject like `chore: cleanup` modified multiple folders.** The tool doesn't try to filter "real" changes from "incidental touches" — if files in the folder changed, the version bumps. This is intentional: the alternative (heuristic filtering) misses real bugs.

3. **A typo in the folder path** — extremely rare, but worth checking.

If the bump is wrong, you have two options:
- **Revert the irrelevant commit's effect on the folder** (`git checkout <commit>^ -- src/Koan.Foo/<file>`)
- **Hand-edit `versions.props`** to override the computed version. Use sparingly — the auto-generation will reassert next run unless you re-pin via another commit.

## Scenario 7 — Recovering from a botched release

If `New-Release.ps1` got partway through and you need to back out:

```pwsh
# Delete the local tag
git tag -d release/v0.7.1

# Delete the remote tag (if you pushed it)
git push origin --delete release/v0.7.1

# Regenerate versions.props if it was committed prematurely
git revert <commit-with-bad-versions.props>
```

If `dotnet nuget push` already published some packages, you can't unpublish from nuget.org — but you CAN list them as unlisted via the NuGet UI. Pin the bad versions in any next release to avoid downstream confusion.

## How the system actually works (read once, then ignore)

**Files**:

```
Directory.Build.props              # No version logic here (must run before csproj sets KoanPackageKind)
Directory.Build.targets            # Imports build/versions.props + resolves <Version> based on KoanPackageKind
build/kernel-manifest.txt          # Authoritative list of kernel package paths (committed; edited by humans)
build/versions.props               # Generated; one PropertyGroup per periphery package + KoanKernelVersion
scripts/versioning/
  Update-Versions.ps1              # Main computation: reads git log per folder, applies conventional-commit rules, writes versions.props
  Show-VersionStatus.ps1           # Read-only wrapper around Update-Versions.ps1 -DryRun
  New-Release.ps1                  # Tags + optionally pushes + optionally publishes NuGet
  Migrate-CsprojToTwoTier.ps1      # One-shot migration tool (kept for reproducibility)
```

**MSBuild evaluation order** (matters for understanding why imports are where they are):
1. SDK targets load
2. `Directory.Build.props` runs (early — too early to read `KoanPackageKind`)
3. csproj body runs, including `<KoanPackageKind>Kernel|Periphery</KoanPackageKind>`
4. `Directory.Build.targets` runs (now `KoanPackageKind` is set; imports `versions.props`; resolves `<Version>`)
5. NuGet pack reads `<Version>` from MSBuild properties

The kernel case is straightforward: `Directory.Build.targets` sees `KoanPackageKind=Kernel` and sets `<Version>` to `$(KoanKernelVersion)`.

The periphery case relies on `versions.props` having a `PropertyGroup` block conditioned on `MSBuildProjectName == 'X' AND KoanPackageKind == 'Periphery'`. When the import runs (after step 3), the condition evaluates true for exactly one block, and `<Version>` is set.

**Conventional-commit detection**:

`Update-Versions.ps1` runs `git log <reference-tag>..HEAD --format=%s -- <folder>` for each periphery folder. The commit subjects are scanned in order:
- First subject matching `!:` or `BREAKING CHANGE` → major
- Else first subject matching `^feat(\(...\))?:` → minor
- Else (any commits found) → patch
- No commits → no bump

**Reference tag**: most recent `release/v*` tag by version sort. The script falls back to "no reference" (bootstrap mode) and freezes the current state if no tags exist.
